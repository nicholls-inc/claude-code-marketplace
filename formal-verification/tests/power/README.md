# Power — K3 smoke test for the Phase 3b-β Lean pipeline

Self-contained fixture exercising `/informal-spec` → `/lean-spec` → `/lean-impl` → `/correspondence-review` → `/drt-oracle` against a deliberately-bugged production implementation. Demonstrates that the DRT harness catches the planted divergence with a witness — the K3 kill criterion in the Phase 3b-β plan.

## Layout

```
formal-verification/
├── specs/power_informal.md                         # Step 1: prose spec, signed off
├── lean/CrosscheckModel/Power.lean                 # Steps 2 + 3: spec stub + functional model
├── correspondence/Power.md                         # Step 4: classification (verdict: READY)
└── tests/power/
    ├── README.md                                   # this file
    ├── drt_harness.py                              # Step 5: DRT driver
    ├── oracle_reference.py                         # Lean-faithful Python reference oracle
    ├── sut/power.py                                # production SUT with planted off-by-one bug
    └── drt_report.md                               # generated by drt_harness.py
```

## Run

```bash
python formal-verification/tests/power/drt_harness.py
```

Exits 1 when divergences are found (the smoke's expected outcome — the SUT is buggy by design).

## Smoke-test scope note

The Lean image was not built in this session because the Claude harness sandbox cannot write to `~/.docker/buildx/activity/`. The harness therefore uses `oracle_reference.py` — a Python implementation that mirrors `CrosscheckModel.Power.power`'s recursion exactly — as the smoke-time oracle. This exercises:

- The five-step pipeline as a sequence of artefacts (informal spec → Lean stub → Lean impl → correspondence → DRT)
- The harness logic (random input generation, dual execution, divergence comparison, shrink minimisation, witness reporting, four-class taxonomy)
- The catch behaviour (planted off-by-one bug surfaces with a small witness)

It does **not** exercise:
- The Lean MCP tools (`lean_check`, `lean_run`) on the Power module
- `lake build` of `Power.lean` against Mathlib

To run a full Lean-side smoke once the image is built:

1. `cd crosscheck && ./scripts/build-lean-docker.sh` (10–30 min on first build).
2. Add a `PowerRunner.lean` file at `formal-verification/lean/CrosscheckModel/PowerRunner.lean` exposing `main : IO Unit` that reads JSON `{"base": Nat, "exp": Nat}` from stdin and prints the result.
3. Wire the harness's `--oracle lean-runner` branch to invoke the runner via `lake exe`.
4. Re-run the harness — the bug should still be caught, and the oracle source is now the actual Lean def.

## Why the planted bug

`sut/power.py` uses `for _ in range(exp + 1)` instead of `for _ in range(exp)`. This adds one extra multiplication, so the SUT computes `b^(n+1)` for `n > 0` while the oracle computes `b^n`. Witnesses exist at any `(base, exp)` with `base != 1` and `exp > 0`; the smallest minimised witness is `(base=2, exp=1)` returning `4` instead of `2`.

This is exactly the class of bug Cedar's bug taxonomy (Disselkoen et al. 2024) shows DRT catches well — a "general implementation bug" (D7a), not an authorization-specific rule-interaction bug. The smoke evidences DRT-as-technique generalising; it does not evidence VGD-as-methodology generalising broadly (D7b).
